## 1. 핵심 개념 정리

### 1.1 하네스 엔지니어링이란
> "AI에게 잘 일할 환경을 설계해 주는 것"

프롬프트 엔지니어링이 "그때그때 잘 요청하는 기술"이라면,  
하네스 엔지니어링은 "AI가 더 잘 일할 수 있도록 일하는 환경 자체를 설계하는 작업"이다.

**4가지 구성 요소:**

| 요소 | 설명 | 현재 구현체 |
|------|------|------------|
| **헌법 (Constitution)** | 우리 팀은 이렇게 일한다. 규칙 문서 | CLAUDE.md, AGENTS.md |
| **작업 구조 (Work Structure)** | 무엇을 어떻게 만들지. 작업 지시서 | PRP, plan docs, tasks |
| **검증 (Verification)** | 결과가 맞는지 체크하는 기준 | typecheck, tests, CI |
| **실행 루프 (Execution Loop)** | 수정→검증→반복하는 워크플로우 | OMC hooks, skills |

**추가 요소:**
- **저장소 (Repository)**: AI가 모든 맥락을 참조하는 곳 (git repo + docs)
- **참고 자료 (References)**: 복잡한 작업을 위한 레퍼런스 문서

### 1.2 OMC/OMX 개발자 허예찬의 핵심 인사이트
- 클로드 코드라는 리포지토리 코드는 사람이 CLI를 건드려서 짠 코드가 없다 (에이전트들만)
- AI 에이전트를 팀처럼 굴리는 것이 핵심 (도구 이름보다 운영 방식이 중요)
- OMX = 코덱스를 마개조한 것 (oh-my-codex) - 하네스로 순정과 다른 런타임 구현

### 1.3 "말 잘 듣는 AI"의 조건 (MIT 연구 기반)
MIT Media Lab 연구 결과: AI를 쓸수록 멍청해지는 사람 vs 똑똑해지는 사람의 차이
- **잘못된 사용**: AI에 수동적으로 의존 → 인지 부채(cognitive debt) 누적 → 뇌 신경 연결성 약화
- **올바른 사용**: AI를 협력 도구로 사용하면서 자신의 사고를 유지 → 함께 성장

**AI가 잘 일하려면**: 사람도 명확하게 사고하고 방향을 잡아야 한다. 하네스는 양방향이다.

### 2.2 paperclip의 에이전트 운영 패턴
Paperclip은 "AI 에이전트 회사"를 위한 제어 플레인:
- **에이전트 = 직원**: 역할, 예산, 목표, 보고 체계 있음
- **하트비트**: 주기적으로 깨어나 작업 확인 후 실행
- **거버넌스**: 사람이 이사회 역할 (승인/거부/감사)
- **비용 제어**: 에이전트별 토큰 예산, 한도 초과시 자동 정지
- **목표 정렬**: 회사 미션 → 팀 목표 → 에이전트 역할 → 태스크

### 2.3 현재 문제점
- 프로젝트마다 CLAUDE.md를 개별적으로 만들어야 함
- 에이전트 설정이 프로젝트별로 파편화됨
- 검증 기준이 프로젝트마다 다름
- 실행 루프가 자동화되지 않음 (수동 트리거)
- Paperclip과 OMC/gstack이 따로 동작

### 3.2 collar 프로젝트의 역할
**collar** = 목줄 (AI에 하네스를 다는 도구)

collar는 다음을 제공하는 **유니버설 하네스 프레임워크**:

1. **프로젝트 초기화 스크립트** (`collar init`)
   - 프로젝트 타입 감지 (Node.js, Python, Rust, etc.)
   - 적절한 헌법(CLAUDE.md) 템플릿 생성
   - AGENTS.md 생성
   - 검증 워크플로우 설정

2. **헌법 템플릿 라이브러리**
   - 언어/프레임워크별 CLAUDE.md 템플릿
   - 역할별 에이전트 정의
   - 공통 규칙 + 프로젝트 특화 규칙

3. **검증 루프 자동화**
   - 언어별 검증 명령어 표준화
   - 실패시 재시도 로직
   - Paperclip에 상태 보고

4. **Paperclip 통합**
   - 태스크 완료시 자동 검증 후 보고
   - 에이전트 신뢰 점수 연동
   - 비용 추적

## 4. 구현 계획

### Phase 1: 하네스 표준 문서 (1주)
목표: 모든 프로젝트에 적용할 수 있는 표준 헌법 템플릿 제작

- [ ] `collar/templates/CLAUDE.md.base` - 공통 헌법
- [ ] `collar/templates/CLAUDE.md.nodejs` - Node.js 특화
- [ ] `collar/templates/CLAUDE.md.python` - Python 특화
- [ ] `collar/templates/AGENTS.md.base` - 에이전트 가이드 템플릿
- [ ] `collar/templates/AGENTS.md.team` - 팀 에이전트 패턴

### Phase 2: collar init CLI (2주)
목표: 새 프로젝트에 하네스 즉시 적용

```bash
cd my-project
collar init
# → 프로젝트 타입 감지
# → 적절한 CLAUDE.md, AGENTS.md 생성
# → .claude/agents/ 폴더 및 기본 에이전트 생성
# → 검증 스크립트 설정
```

### Phase 3: Paperclip 연동 (3주)
목표: collar로 관리되는 프로젝트를 Paperclip이 오케스트레이션

- collar 에이전트가 Paperclip API를 통해 태스크 체크아웃
- 완료시 검증 후 자동 보고
- 신뢰 점수에 따른 자율성 조정

### Phase 4: 메모리/학습 시스템 (4주)
목표: 프로젝트별 학습이 글로벌로 축적되는 시스템

- gstack learnings 시스템 확장
- 성공 패턴 자동 추출 및 템플릿화
- 실패 패턴 자동 차단 (hooks)

**패턴 악화 추세:** March → May, 동일 패턴이 점점 심해짐

### 9.2 gstack이 못 잡은 이유

| gstack의 현재 설계 | 실제 빠진 것 |
|-------------------|------------|
| `<verification>` 규칙 문서에만 있음 | 실행을 강제하는 체크포인트 없음 |
| "완료" 언어 규정 없음 | STARTED/TESTED/VERIFIED 구분 없음 |
| 요구사항 모호성 가드 없음 | "효율적으로" 같은 추상 표현 통과됨 |
| 다음 단계 명시 규칙 없음 | 사용자가 계속 "이어서"를 입력해야 함 |
| 재발 버그 체크 없음 | 새 수정 전 이전 수정 생존 확인 없음 |

### 9.3 collar가 추가한 설계 (v2 템플릿 반영 완료)

1. **완료 3단계**: STARTED / TESTED / VERIFIED — 어느 단계인지 명시 필수
2. **증거 기반 완료**: 빌드 출력, 테스트 결과, 실행 스크린샷 없이 done 금지
3. **모호한 표현 차단**: "효율적으로" 등 사전 정의 목록 → 코드 전 구체화 요청
4. **다음 단계 명시**: 모든 작업 후 "다음: X. 진행할까요?" 패턴 강제
5. **재발 버그 체크**: "또 안됨" 보고 시 이전 수정 생존 확인 우선
